# 命令执行策略（防止等待人工授权）

## 允许的命令类型（优先使用）

- 本地文件操作：读取/写入项目文件、生成代码、更新配置。
- 本地构建与测试：`mvn test` / `npm test` / `pnpm test` / `gradle test` 等普通测试命令。
- Git 本地命令：`git status`、`git diff`、`git add`、`git commit`（遵守本 Spec 的 Git 规范）。

## 限制/禁止的命令类型（默认不要执行，防止卡在权限）

- 禁止执行需要高权限或可能被拦截的命令：例如全局安装、系统配置修改、`sudo`、`docker system prune` 等。
- 禁止执行破坏性 Git 命令：`git push --force`、`git reset --hard`、`git rebase` 等。
- 对任何被权限系统拒绝的命令，不要反复重试，不要等待人工确认，直接：
  - 在 `tasks.md` 中记录一个 TODO（标注需要人工/管理员执行的命令和原因）；
  - 在 `requirements.md` / `design.md` 相关章节里备注"此步骤需人工运维执行"。

## 行为要求

- 遇到命令权限不足或外部依赖不可用时：
  - 立刻停止该命令方向的尝试；
  - 用中文说明"已跳过该命令，并在文档/任务中留下 TODO"，继续完成其余能做的代码与文档工作。

# 离线工作优先级（适配无人值守场景）

- 在无法确定命令是否有权限执行时，你应优先：
  - 完成代码实现、单元测试代码、集成测试代码的编写；
  - 更新 `requirements.md` / `design.md` / `tasks.md`；
  - 在注释或文档中标明"待人工执行的环境/部署命令"。
- 不要求你一定要成功跑完所有测试或部署命令，但要求你：
  - 写出可直接运行的命令；
  - 在命令可能失败/需权限时，提前在文档里标注"需运维/管理员手工执行"。

---

# 工作模式说明（无人值守优化）

- 假设大部分时间用户不在电脑前，无法实时点权限/确认。
- 因此，当你需要做可能被权限拦截的操作时，应遵循：
  - "**能写代码就先写代码，能写文档就先写文档**"，
  - 不能执行的命令一律记为 TODO，而不是等用户。
- 你的目标是：
  - 即使用户不在线，也能让项目代码/文档向前推进到"人来只需要跑一两条命令就能验证和部署"的状态。

# 任务状态与执行记录（强制）

## 任务状态字段

- 每个 `TASK-*` 必须包含 `status` 字段：`TODO` / `IN_PROGRESS` / `DONE` / `BLOCKED`。
- 当你开始执行某个任务时，必须将其 `status` 更新为 `IN_PROGRESS`。
- 当任务完成（代码/测试/Git 提交已按规范完成）时，必须将其 `status` 更新为 `DONE`。
- 如果任务因权限/外部依赖问题无法继续，必须将其 `status` 更新为 `BLOCKED`，并在 `blockedReason` 中写明原因和需人工操作的命令或动作。

## 执行记录（Execution Log）

- 每个 `TASK-*` 必须有一个简短的 `executionNotes`（或类似字段），在任务执行完毕（或被 BLOCKED）时更新：
  - 简述已完成的工作（代码/测试/文档/Git 提交）
  - 若 BLOCKED：列出具体阻塞点（命令/权限/外部系统）和建议的人工处理步骤
  - 要求使用中文简要说明，方便人类接手。

---

# Task 小结与离线可读性（强制）

- 每当一个 `TASK-*` 执行结束（无论是 `DONE` 还是 `BLOCKED`），你必须：
  - 在 `tasks.md` 中更新该任务的 `status` 与 `executionNotes`；
  - 在项目根目录的 `progress-log.md` 追加一条记录（附时间戳、TASK 编号、简短中文小结）。
- `progress-log.md` 每条记录建议格式：
  - `[YYYY-MM-DD HH:mm] TASK-<编号> - DONE/BLOCKED - <一句话中文总结>`
  - 若 BLOCKED：简单写明"因 <原因> 暂停，待人工执行：<命令/操作>"。
